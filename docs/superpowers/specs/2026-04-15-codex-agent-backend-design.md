# Codex Agent Backend Design Spec

## Overview

Autotune currently shells out only to the `claude` CLI even though the internal agent boundary is already abstracted behind the `autotune_agent::Agent` trait. We want `codex` to be a drop-in backend alternative that can be selected globally or per role, and mixed with Claude in the same task.

The goal is to support configurations like:

```toml
[agent]
backend = "claude"

[agent.research]
backend = "codex"

[agent.implementation]
backend = "claude"

[agent.init]
backend = "codex"
```

without changing the rest of the tune loop, state machine, or XML protocol.

## Architecture

### Backend Factory in `autotune`

Backend selection should move out of `main.rs`'s current hard-coded `ClaudeAgent` construction and into a small factory that returns `Box<dyn Agent>` for a requested backend name.

The factory resolves backend per role using existing config shape:

1. role-specific `agent.<role>.backend`
2. fallback to top-level `agent.backend`
3. fallback to config default (`"claude"`)

This keeps backend choice where it belongs: at the call site that knows whether it is building a research, implementation, or init agent.

### New Backend: `autotune_agent::codex::CodexAgent`

Add a new module alongside the existing Claude implementation:

```text
crates/autotune-agent/src/
├── claude.rs
├── codex.rs
└── lib.rs
```

`CodexAgent` implements the same `Agent` trait as `ClaudeAgent`:

- `spawn()`
- `send()`
- `spawn_streaming()`
- `send_streaming()`
- `backend_name()`
- `handover_command()`
- `grant_session_permission()`

Like `ClaudeAgent`, it stores per-session context in memory so resumed turns inherit:

- working directory
- model
- max turns, if mapped
- effective logical permission set
- any backend-specific config derived from those permissions

That session memory is required so a resumed Codex session behaves consistently across `spawn()` and `send()`, and so mixed-backend runs can resume the right backend cleanly.

## Codex CLI Contract

Autotune should target the local `codex` CLI's non-interactive interface:

- initial turn: `codex exec ...`
- resumed turn: `codex exec resume <session-id> ...`
- model selection: `--model`
- working directory: `-C`
- structured stream output: `--json`

The agent prompt should be passed the same way as Claude today: the full prompt string generated by Autotune is the backend input. No protocol changes are needed at the Autotune layer.

### Session IDs

`AgentResponse.session_id` should store the Codex session identifier returned by the CLI. `AgentSession.backend` already exists and should continue to be the source of truth when resuming persisted sessions.

### Handover Command

`handover_command()` for Codex should emit a concrete resume command, for example:

```bash
codex resume <session-id>
```

This mirrors the current Claude handoff behavior while keeping the printed command backend-specific.

## Permission Model Mapping

The hardest part of drop-in compatibility is that Claude and Codex expose different control surfaces.

### Claude Today

Claude accepts explicit tool permissions:

- allow tool
- allow tool scoped to a path
- deny tool

Autotune uses that to sandbox the implementation agent tightly and to widen permissions later via `grant_session_permission()`.

### Codex Needed Behavior

Codex exposes sandbox and approval controls rather than the same named-tool allowlist model. The design should preserve Autotune's current external behavior by translating Autotune's logical `ToolPermission` set into Codex CLI options.

The mapping should be:

- read-only planning/init configurations map to a read-only or no-shell-capable Codex configuration
- implementation configurations map to workspace-write with the worktree as the writable root
- extra scoped writable paths map to `--add-dir` when needed
- denied shell access maps to a Codex configuration that does not permit unsandboxed command execution

This translation lives inside `CodexAgent`, not in calling crates.

### Runtime Permission Grants

Autotune's approval flow must still work for Codex-backed sessions. Since Codex does not expose the same runtime tool mutation flags as Claude, `grant_session_permission()` should update the stored logical permission set for the session, then ensure the next resumed `send()` is launched with regenerated Codex CLI flags that reflect the widened permissions.

This preserves the CLI's behavior:

1. agent requests more access
2. Autotune approves it
3. later turns run with broader permissions

even if the Codex CLI implements that broader access through sandbox flags rather than named tools.

## Streaming Events

Codex should support the same high-level event surface Autotune already emits:

- `AgentEvent::ToolUse`
- `AgentEvent::Text`

The backend adapter is responsible for decoding `codex exec --json` output into those events. If Codex emits more event types than Autotune currently needs, the adapter can ignore the extras.

The important behavior is parity with Claude callers:

- non-streaming methods still work
- streaming methods power interactive status output
- final message extraction remains robust even if intermediate JSON events vary

## Mixed Backend Operation

Mixed-backend support is a first-class requirement, not a follow-up:

- research can be Codex while implementation is Claude
- init can use Codex without affecting run-time planning
- resumed task state must keep using the backend recorded on the session

This means backend selection must happen independently for each agent construction site, and any code that resumes an existing session must use `AgentSession.backend` rather than assuming today's process-wide agent instance is Claude.

## Error Handling

Backend-specific failures should remain actionable at the CLI surface:

- unknown backend name -> clear config error listing supported backends
- Codex CLI missing -> command failure naming `codex`
- malformed JSON/event output -> parse failure naming Codex output
- resume without stored session context -> explicit backend/session error

Error text should identify the backend so mixed-backend debugging is practical.

## Testing

This feature needs coverage at three levels.

### `autotune-agent` unit and backend tests

- `CodexAgent` arg construction for spawn and resume
- response parsing and streaming event parsing
- session context preservation across `spawn()` + `send()`
- permission grant behavior widens next-turn config
- handover command formatting

Use a fake `codex` executable in tests, mirroring the current Claude backend harness style.

### `autotune-config` and protocol tests

- backend defaults remain `"claude"`
- role-level backend overrides still parse
- protocol/config structures accept `"codex"` anywhere `"claude"` is accepted today

### `autotune` integration/scenario tests

- global backend set to `codex`
- mixed role config (`research=codex`, `implementation=claude`, and the reverse)
- persisted sessions continue with the correct backend
- init path can select Codex from config defaults

Mock-backed tests stay useful for state-machine behavior, but backend-selection paths need explicit coverage so `main.rs` no longer silently assumes Claude.

## Non-Goals

- Adding an OpenAI API-native backend inside Rust
- Redesigning the `Agent` trait around a new capability system
- Changing Autotune's XML protocol
- Making Codex semantics identical to Claude at the CLI-flag level

The contract is behavioral compatibility at the Autotune layer, not identical subprocess arguments.
